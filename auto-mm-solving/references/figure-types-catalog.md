# Figure types catalog — 数模 contests

A catalog of the figure types that show up in 数模 papers, with the **production channel** for each. Stage 2 reads this when classifying a figure in `brief.md`. The three channels:

| Channel | When to use | Folder layout |
|---|---|---|
| **`data`** | The figure is driven by real numbers from `stage2_solving/runs/<exp_id>/`. Generated by running `plot.py` (matplotlib / seaborn) or `tikz.tex` against the data. | `brief.md` + `prompt.md` (code) + `source/plot.py` + `source/data_used.csv` + `output.pdf` + `self_check.md` |
| **`schematic`** | The figure illustrates a concept or workflow with no real-data dependency. Generated by AI image-gen (DALL-E / Imagen / Midjourney / Nano Banana) for aesthetic schematics, or by TikZ for label-heavy diagrams. | `brief.md` + `prompt.md` (image or TikZ) + `source/image_request.json` *or* `source/tikz.tex` + `output.pdf|png` + `self_check.md` |
| **`sourced`** | The figure must depict a real-world object (a specific city map, a policy document, a physical device, satellite imagery, historical photo). **AI must not generate this** — fabricated geography / artifacts are a research-integrity failure. | `brief.md` + `search_queries.md` + `sources/` (candidates + .meta.json) + `chosen.png` + `attribution.md` + `self_check.md` |

If a figure could plausibly be in two channels, the rule is: real numbers → `data`; pure concept → `schematic`; real-world reference → `sourced`.

## Channel `data` — driven by real numbers

### Trend & comparison
| Type | Tool | Used for |
|---|---|---|
| **Line plot (single / multi series)** | matplotlib | Convergence curves, time-series forecasts, trend by parameter |
| **Convergence curve** | matplotlib | ALNS / SA / TS objective per iteration; useful for showing the algorithm earns its cost |
| **Bar chart (grouped / stacked)** | matplotlib | Method comparison, cost breakdown by category, ablation deltas |
| **Histogram / density** | matplotlib + seaborn | Distribution of a quantity (residuals, demand, time-window slack) |
| **Box plot / violin** | seaborn | Spread + IQR + outliers (per method, per scenario) |
| **Error-bar plot** | matplotlib | Mean ± CI across seeds; ablation with uncertainty |

### Multi-dimensional
| Type | Tool | Used for |
|---|---|---|
| **Scatter plot** | matplotlib | Data shape, relationships, residuals vs predicted |
| **Scatter matrix / pair plot** | seaborn | Quick joint diagnostic across features |
| **Heatmap** | seaborn | Sensitivity 2D sweep, correlation matrix, demand heatmap by zone × hour |
| **Pareto front** | matplotlib | Multi-objective trade-offs (cost vs emission, accuracy vs latency) |
| **Radar / spider chart** | matplotlib | Per-method or per-scenario multi-criteria comparison |

### Spatial / scheduling
| Type | Tool | Used for |
|---|---|---|
| **Route map** | matplotlib + folium / geopandas | VRP solutions, delivery paths, customer→vehicle assignment |
| **Spatial scatter** | matplotlib + geopandas | Customer locations, sensor distribution, event geography |
| **Gantt chart** | matplotlib + matplotlib.patches | Job / vehicle / machine scheduling |
| **Spacetime trajectory** | matplotlib | Position × time for vehicles or agents |
| **Network / graph** | NetworkX + matplotlib | Graph topology, flow, centrality, community |

### Inference / model output
| Type | Tool | Used for |
|---|---|---|
| **ROC / PR curve** | sklearn + matplotlib | Classifier diagnostics |
| **Confusion matrix** | sklearn + matplotlib | Classifier diagnostics |
| **Calibration / reliability** | sklearn + matplotlib | Probabilistic forecast diagnostics |
| **Feature importance** | matplotlib + LightGBM/XGBoost API | Tree-model post-hoc explanation |
| **Attention weights** | seaborn (heatmap) | NN interpretability (with the Rule 4 caveat about meaningful uncertainty) |

### Validation
| Type | Tool | Used for |
|---|---|---|
| **Gap vs instance size** | matplotlib | Reporting heuristic Gap to exact across N |
| **Convergence to optimum** | matplotlib | Branch-and-bound / column generation progress |
| **Cost breakdown stacked area** | matplotlib | Where the total cost goes as a parameter sweeps |

## Channel `schematic` — concept illustrations

### Algorithm / system structure
| Type | Tool | Used for |
|---|---|---|
| **Model framework flowchart** | TikZ | The headline §3 figure: input → preprocessing → MILP → ALNS → validation → output |
| **System architecture** | TikZ | Components + arrows, no real numbers |
| **Algorithm pseudocode block** | LaTeX `algorithm2e` (not a figure but illustrated like one) | Inline algorithmic structure |
| **State-transition diagram** | TikZ | RL / dynamic programming state machines |
| **Decision tree (illustrative)** | TikZ | When you want to show the structure, not real splits |

### Mathematical concept
| Type | Tool | Used for |
|---|---|---|
| **Geometric setup** | TikZ | "Here is the green zone, here is the constraint boundary" |
| **Attention mechanism diagram** | TikZ | QKV multi-head architecture (illustrative) |
| **Optimization landscape sketch** | TikZ | Local min vs global min, gradient direction |
| **Bayes network / probabilistic graphical model** | TikZ | Latent variables and observed nodes |

### High-level conceptual
| Type | Tool | Used for |
|---|---|---|
| **Conceptual cover / metaphor** | AI image-gen | Rare; only when the paper benefits from a "scene" |
| **Stakeholder map** | TikZ (or AI as last resort) | Policy problems with many actors |
| **Timeline / phase diagram** | TikZ | "Phase 1: data; Phase 2: model; Phase 3: deploy" |

**Default for schematic**: prefer TikZ when the diagram has text labels (text rendering in image-gen is unreliable). Use AI image-gen only when the aesthetic dominates and labels are minimal / external.

## Channel `sourced` — must come from the real world

These figures must NOT be generated by AI. Fabricating a map of "the green zone" when a real green zone exists is a research-integrity failure (similar in spirit to fabricated citations — Rule 3).

| Type | Where to find | Used for |
|---|---|---|
| **Real region / city map** | OpenStreetMap, Google Maps, 国家地理信息公共服务平台 (天地图), 高德 / 百度 Maps | Showing the actual area the problem refers to |
| **Satellite imagery** | NASA Earth, Copernicus, Bing Maps Aerial, USGS Earth Explorer | Land-use, terrain, post-disaster imagery |
| **Specific device / product photo** | Manufacturer website, Wikimedia Commons | Equipment named in the problem (vehicles, sensors, machinery) |
| **Historical photo** | Wikimedia Commons, archive.org, Library of Congress, 中国国家图书馆 | Background context where the problem is historical |
| **Official policy / regulation screenshot** | The issuing authority's website (state council, ministry, industry body) | Citing a specific clause that bounds the model |
| **Official dataset documentation** | Source publisher's page (CDC, World Bank, NBS 国家统计局, COMAP problem page) | Documenting the data lineage |
| **Standards / specification cover** | Issuing standards body (ISO, GB/T, ASTM) | Citing a standard the problem invokes |
| **Public infrastructure data** | OpenStreetMap, GIS portals, transit authority sites | Network topology that already exists in the real world |

**The discipline**: every `sourced` figure must record (a) the original URL, (b) the retrieval date, (c) the license / fair-use rationale, (d) the citation form for `references.bib`. The `attribution.md` file inside the figure folder holds all four.

**Forbidden practices**:
- "AI-cleanup" of a real photo until it loses traceability to the original.
- Screenshot of someone else's paper figure (this is copyright + plagiarism; cite the paper and reproduce with permission, or re-derive from the same data).
- Using a misleading proxy (e.g. a different city's map labeled as the problem's city) — same integrity violation as fabricating data.

## Quick classifier (when in doubt)

Ask in order:

1. **Does the figure depend on real numbers from your experiments?** → `data`.
2. **Does the figure depict a specific real-world place / object / document?** → `sourced`.
3. **Otherwise** (the figure illustrates an idea, an algorithm, a structure) → `schematic`.

If two channels could apply: `sourced` wins over `schematic` (real wins over illustrative); `data` wins over `schematic` (numbers win over concept).

## Typical figure budget for a 25-page paper

Approximate distribution. Adjust per problem.

| Channel | Count | Examples |
|---|---|---|
| `data` | 6-9 | Convergence, results bar chart, ablation, sensitivity sweep, Pareto, validation Gap, residual diagnostic |
| `schematic` | 1-3 | Model framework flowchart, algorithm overview, optional concept sketch |
| `sourced` | 0-3 | Region map, policy screenshot, dataset documentation — only when the problem mentions specific real-world entities |

Total 8-12 figures is the sweet spot; > 15 invites suspicion of padding.

## What auto-mm-solving does with each channel

- **`data`**: Step 5 of `auto-mm-solving/SKILL.md`. Brief → matplotlib/TikZ prompt → `python plot.py` → self-check.
- **`schematic`**: Step 5 of `auto-mm-solving/SKILL.md`. Brief → TikZ or image-gen prompt → render → self-check.
- **`sourced`**: Step 5 of `auto-mm-solving/SKILL.md`, sub-procedure in `figure-sourcing.md`. Brief → search queries → download candidates → choose + cite → self-check.

All three channels share the same `<fig_id>/` folder pattern under `stage2_solving/figures/`; only the internal files differ. Stage 3 reads `<fig_id>/status` and refuses to `\includegraphics` anything not marked `ready`.
